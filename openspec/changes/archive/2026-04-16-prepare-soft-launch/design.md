## Context

The site runs Next.js 16.1.6 with the new `proxy.ts` middleware convention (instead of `middleware.ts`), uses `next-intl` for routing/i18n across `pl`, `en`, `de`, `it`, and is deployed on Vercel (free plan). The current `src/proxy.ts` chains a single call to `next-intl`'s middleware with the matcher `["/((?!api|_next|_vercel|assets|.*\\..*).*)"]`.

Until now, the contact page (`/[locale]/contact`) has assumed a single business address — stored as one multi-line string in `messages/{locale}.json` under `contact.info.address`. The map below it (`ContactMap.tsx`) is a static PNG screenshot wrapped in a Google Maps link. The contact info is consumed in two places: `ContactInfo.tsx` and `Footer.tsx`.

The mobile navigation in `Navbar.tsx` renders a hamburger button below `md` breakpoint, opening a full-screen overlay that lists nav links only. The language switcher (`<div class="hidden ... md:flex">`) is desktop-only, so on mobile there is no way to change language at all.

This change prepares the site for a soft-launch period where stakeholders preview the production deployment before public release. The gate is intentionally temporary and intentionally simple.

## Goals / Non-Goals

**Goals:**
- Production site is unreachable to anyone without a single shared password.
- The two physical Drovar locations are both visible on the contact page and in the footer, with consistent map visuals.
- Maps are reproducible (config + script), not artisanal screenshots.
- Mobile users on every locale can change language without leaving the site.
- The gate can be removed by deleting one env var, with no code rollback.

**Non-Goals:**
- Per-user passwords, account management, or login UI branding (HTTP Basic prompt is acceptable).
- Vercel Deployment Protection (paid feature, not on the free plan).
- Cloudflare Access or any external auth provider.
- Interactive maps (pan/zoom). Static images are sufficient — clicks open the user's preferred maps app.
- Per-location phone numbers or emails. Both sites share company contact details.
- Sanity preview / draftMode bypass. The CMS lives in a separate project and isn't accessed through this site's routes.
- Sitewide content audit, copy refinement, or visual polish — out of scope.
- Replacing `next-intl` middleware behavior or changing the matcher beyond what the gate requires.

## Decisions

### 1. HTTP Basic Auth in `proxy.ts` over Vercel Deployment Protection

**Choice**: Add a 401 + `WWW-Authenticate: Basic` response in the existing `proxy.ts` before delegating to `handleI18n`.

**Why**: Vercel Deployment Protection is a Pro-tier feature; the project is on the free plan. HTTP Basic Auth is ~15 lines of code, requires no UI work, and is fully removable by deleting one env var. The gate is temporary, so investing in a branded login page is overkill.

**Alternatives considered**:
- *Custom login page + signed cookie session* — better UX but ~150 LoC, brings a session library (`jose` or `iron-session`), and is wasted work if removed in weeks.
- *Cloudflare Access* — requires putting Cloudflare in front of Vercel, an infrastructure change for a temporary gate.

**Implementation note**: The expected value is `process.env.SITE_AUTH_BASIC`, the base64 of `user:password`. The check uses `crypto.timingSafeEqual` on equal-length buffers to avoid timing leaks. When the env var is missing or `NODE_ENV !== "production"`, the gate is bypassed entirely.

### 2. Single env var as the on/off switch

**Choice**: Presence of `SITE_AUTH_BASIC` controls the entire gate (auth + robots + sitemap behavior).

**Why**: One toggle, one rollback. Removing the var on Vercel:
- Disables the auth check (gate code returns early)
- Restores the original `robots.ts` rules (allow `/`)
- Restores the populated sitemap

This avoids leaving stale "private" SEO behavior after launch. No re-deploy required after Vercel restarts the function.

### 3. Build-time map generation, not runtime

**Choice**: Both location maps are generated by `scripts/generate-location-maps.ts`, run manually (or in CI) when addresses change. The PNGs are committed to `public/images/` and served as static assets.

**Why**:
- Addresses are extremely stable — changing them is a deliberate event.
- No Azure subscription key needed in the Vercel runtime — key only lives in `.env.local` for the developer running the script.
- Zero runtime cost, zero latency, zero new dependency.
- Matches the existing pattern (`map-wieliczka.png` is already a committed static asset).
- The script geocodes addresses via Azure's Search Address API, so coords don't have to be hardcoded.

**Alternatives considered**:
- *Server-side fetch on each page render* — ships the key to Vercel, adds runtime latency, requires caching policy.
- *Azure Maps Web SDK (interactive map)* — ~250KB JS, cookie/GDPR considerations, ships the key to the browser. Massive overkill for two pin locations.
- *OpenStreetMap static tile* — works without auth but visual style differs from current map and doesn't match the user's stated preference for Azure.

### 4. Spec-driven location data shape

**Choice**: Replace the `contact.info.address` string with a `contact.info.locations` array of two entries, each with:
- `roleLabel` (translated): "Siedziba" / "Office, Warehouse, Production"
- `companyName` (untranslated): exact legal/colloquial name
- `street` (untranslated): street + number
- `cityLine` (untranslated): postal code + city + country if non-PL
- `mapsUrl` (untranslated): external maps app URL
- `mapImage` (untranslated): path to the generated PNG

**Why**: Keeps localized text minimal (only the role label needs translating), avoids accidental address divergence between locales, and makes the data shape explicit for the `<LocationMap>` component to consume. The locations live in `src/lib/locations.ts` as a typed const, with role labels pulled from translations at render time.

**Alternatives considered**:
- *Two separate translation keys* (`address1`, `address2`) — flat but doesn't scale and loses semantic meaning.
- *Sanity-managed addresses* — overkill for two static addresses that change ~once a decade.

### 5. Mobile language switcher inside the hamburger overlay

**Choice**: Render a horizontal row of language chips below the nav links in the existing mobile overlay, separated by a thin divider.

**Why**: The hamburger overlay already exists; adding a block at the bottom is the lowest-friction integration. Keeps the navbar header clean (logo + hamburger only) which is critical at 375px where horizontal space is scarce. Matches the established pattern of "everything navigation lives in the overlay."

**Alternatives considered**:
- *Always-visible chip beside the hamburger* — eats horizontal space, requires a popover for 4 options, more code.
- *Bottom-sheet language picker triggered from a separate icon* — more UI, more components, no clear benefit for a 4-option list.

### 6. Server Actions stay gated, `/api/revalidate` stays open

**Choice**: The existing matcher `["/((?!api|_next|_vercel|assets|.*\\..*).*)"]` already excludes `/api`. The contact form is a Server Action (`actions.ts`), which POSTs to its hosting page route — therefore it goes through the proxy and is automatically gated.

**Why**: The Sanity revalidation webhook must remain reachable from Sanity's servers regardless of the gate, and it's already protected by its own webhook secret. The contact form, on the other hand, should not be submittable while the site is gated — exactly what we get for free.

## Risks / Trade-offs

- **Static `/assets` and `/_next/static` are not gated.** Anyone with a deep link can fetch `/assets/logo.svg` or `/_next/static/...` directly. *Acceptable*: the gate is for content discoverability, not data exfiltration. The actual content (pages, copy, images shown in context) requires authentication.
- **HTTP Basic Auth has no logout flow.** Users must close the browser or clear credentials. *Acceptable*: the audience is small (stakeholders, internal team), and the gate is temporary.
- **Browser-saved Basic credentials may leak via screenshots/screen-shares of the URL bar.** Some browsers historically rendered `user@host` in the bar; modern Chrome/Safari/Firefox do not. *Acceptable* given the threat model.
- **Build-time map generation requires a manual script run when addresses change.** *Acceptable*: addresses change on the order of years. The script has a one-line npm command.
- **Azure Maps key was shared in chat during planning.** *Mitigation*: rotate the key in the Azure portal before the script runs, and apply referer restrictions limited to `drovar.pl` + `localhost`.
- **`robots.ts` and `sitemap.ts` need to read `process.env.SITE_AUTH_BASIC`.** Both are server-evaluated at request time, so they see live env vars — fine on Vercel, no caching pitfall.
- **If Vercel cold-starts make the gate momentarily inconsistent across instances.** *Not applicable*: the gate is stateless, every instance reads the same env var.
- **`useActionState`-based form keeps state in the gated page**. Submitting while gated returns 401 from the page route, which is a server-side redirect to nothing — the user is already authenticated to be on the page, so this is a non-issue in practice.

## MODIFIED Requirements

### Requirement: Static map displays office location
The system SHALL display one static map image per Drovar location on the contact page. Each map SHALL be stored as a static asset in `public/images/`, generated by a build-time script (`scripts/generate-location-maps.ts`) that calls the Azure Maps Render Static Image API with a centered pin at the geocoded coordinates of the address. Each map SHALL be rendered using `next/image` and SHALL be wrapped in a link to the corresponding external maps URL (`mapsUrl`).

#### Scenario: Headquarters map renders
- **WHEN** the contact page loads
- **THEN** a map image showing the Wieliczka headquarters location with a pin is visible below the headquarters address

#### Scenario: Production map renders
- **WHEN** the contact page loads
- **THEN** a map image showing the Myślenice production location with a pin is visible below the production address

#### Scenario: Map image is optimized
- **WHEN** any location map renders
- **THEN** it uses `next/image` with appropriate width, height, and locale-aware alt text

#### Scenario: Tapping a map opens external maps app
- **WHEN** the user taps a map image
- **THEN** a new tab opens at the location's `mapsUrl` (Google Maps for that pin)

### Requirement: Map has inset border styling
Every location map SHALL display with the same inset border treatment used by the previous single-map implementation, consistent with the project's design language.

#### Scenario: Inset border is visible on every map
- **WHEN** any location map renders
- **THEN** it has the project's standard inset border effect

### Requirement: Map is responsive
Each location map SHALL fill the available width of its container and maintain aspect ratio across all viewports.

#### Scenario: Map fills container on mobile
- **WHEN** viewed on a 375px viewport
- **THEN** each map image fills the full width of the contact info column

#### Scenario: Map fills container on desktop
- **WHEN** viewed on a 1280px viewport
- **THEN** each map image fills the width of the left column (`lg:col-span-2`)

## ADDED Requirements

### Requirement: Maps are generated by a reproducible script
The system SHALL provide a Node script `scripts/generate-location-maps.ts` that reads the location list from `src/lib/locations.ts`, geocodes each address via Azure Maps Search Address API, fetches a static map PNG via Azure Maps Render Static Image API, and writes each PNG to the location's `mapImage` path. The script SHALL be runnable via `npm run generate:maps`. The Azure subscription key SHALL be read from `process.env.AZURE_MAPS_KEY` and SHALL never be referenced from runtime application code.

#### Scenario: Script generates both PNGs
- **WHEN** the developer runs `npm run generate:maps` with `AZURE_MAPS_KEY` set
- **THEN** both `public/images/map-wieliczka.png` and `public/images/map-myslenice.png` are written with current map tiles and a pin at the geocoded address

#### Scenario: Script fails clearly without key
- **WHEN** the developer runs `npm run generate:maps` without `AZURE_MAPS_KEY`
- **THEN** the script exits with a non-zero status and a clear error message

#### Scenario: Azure key never reaches runtime
- **WHEN** the production application is built and deployed
- **THEN** `AZURE_MAPS_KEY` is not referenced anywhere outside `scripts/generate-location-maps.ts` and is not required to be set in Vercel environment variables
